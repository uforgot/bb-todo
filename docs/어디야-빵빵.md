# 어디야 빵빵 — 위치 정보 통합 플랜

위치 정보를 두 갈래로 활용한다.
하나는 보이스로 말할 때 페이로드에 좌표를 동봉해 봇이 "어디서 묻는지" 알게 하는 것, 다른 하나는 핸드폰이 Ably 채널로 위치 이벤트를 푸시해 서버에 스택으로 쌓고, 그 신호에 반응해서 봇이 먼저 말 거는 하트비트.

## 목표

- 보이스 발화 시점에 좌표 즉시 동봉 (요청-응답형).
- 백그라운드/포어그라운드에서 위치 변동 이벤트를 Ably로 푸시 → 서버 스택에 누적.
- 하트비트는 랜덤 타이머가 아니라 "위치가 바뀌었다", "한 곳에 오래 머문다" 같은 위치 신호에 반응.
- 좌표는 그대로 모델에 던지지 않고, 등록된 장소 이름으로 변환해 의미 있게.
- 등록되지 않은 좌표는 Google Places `nearby`로 보강. 그것도 안 되면 좌표 그대로.

## 범위 (제외 항목)

- 항상 추적은 아님. 의미 있는 변동(significant location change)만 푸시.
- 가족 폰 트래킹/지오펜스는 별개 프로젝트. 여기는 본인 폰만.
- 카메라/이미지 동반 전송은 별도 항목.

## 두 채널

### A. 보이스 요청 채널 (즉시)

보이스로 말할 때마다 현재 좌표를 같이 보낸다. 봇이 응답할 때 "어디서 묻는지" 안 채로 답.

### B. 하트비트 채널 (Ably 푸시 + 서버 스택)

핸드폰이 이벤트 기반으로 Ably `location` 채널에 좌표 푸시.
서버는 최근 N개를 메모리/SQLite 스택으로 보관. 봇은 그 스택을 보고 의미 있을 때 한 마디.

스택 보관 예시 (최근 24h):
```jsonc
[
  { "ts": "2026-05-15T12:01:00Z", "lat": ..., "lng": ..., "place": "회사" },
  { "ts": "2026-05-15T13:25:00Z", "lat": ..., "lng": ..., "place": "스타벅스 강남" },
  { "ts": "2026-05-15T14:10:00Z", "lat": ..., "lng": ..., "place": "회사" }
]
```

하트비트 발화 트리거 후보:
- 새 장소 도착 (스택에 새 `place`가 들어옴).
- 한 장소에 오래 머무름 (예: 2시간 이상 동일 place).
- 등록된 장소로 복귀 ("집 도착").
- 특정 시간대 + 특정 장소 조합 (점심시간에 회사 → 점심 추천).

## 컴포넌트

### 1. iOS — 위치 권한 + 좌표 캡처

- `Info.plist`: 
  - `NSLocationWhenInUseUsageDescription` — 보이스 대화 시 사용.
  - `NSLocationAlwaysAndWhenInUseUsageDescription` — 하트비트용 백그라운드 위치 변동 감지.
- `Services/LocationService.swift` (신규): `CLLocationManager` 래퍼.
  - `requestLocation()` — 보이스 발화 시 1회성.
  - `startSignificantLocationChanges()` — 하트비트용 저전력 변동 감지.
- `VoiceView` `onAppear`에서 권한 요청. 거부 시 위치 없이 진행.
- `VoiceService.publishRequest` 페이로드에 `location: { lat, lng, accuracy, ts }` 동봉 (채널 A).
- 위치 변동 이벤트 발생 시 Ably `location` 채널로 publish (채널 B).

### 2. 데이터 저장소 — bb-admin이 단일 소유

- `voice-config.json`에는 **places 데이터를 넣지 않는다**. 운영 설정과 섞이는 게 싫음.
- bb-admin에 `places` 전용 스토어 (얼굴 인식 데이터셋이 bb-admin에서 관리되는 것과 같은 패턴).
  - 파일 위치 후보: `bb-admin/data/places.json` 또는 face-admin과 같은 storage layer.
  - 스키마: `{ id, name, lat, lng, radiusM, placeId?, address?, createdAt }[]`.
  - `placeId` — Google Places place_id (등록 시 autocomplete로 받은 정식 id).

### 3. 서버 — 좌표를 이름으로 변환 + 스택 관리

- `voice-bridge.js`가 bb-admin API에서 places 리스트를 받아옴 (캐싱, TTL 짧게).
- 좌표 들어오면 places 리스트와 비교. 반경 안이면 이름으로 치환.
- 매칭 실패 시 `goplaces nearby` 호출 → 가장 가까운 POI 이름.
- 그것도 실패하면 `"위도 X 경도 Y"` 그대로.
- Ably `location` 채널 수신 → SQLite `location_stack` 테이블에 누적 (최근 24h만 유지).
- 스택 변화 감지 로직: 새 place 도착 / 머무름 / 복귀 → 하트비트 큐에 트리거 푸시.

### 4. 등록 UI — bb-admin 페이지 + API

- `bb-admin/src/app/places/` 페이지 신규.
  - 목록 / 추가 / 수정 / 삭제.
  - 추가 시 Google Places autocomplete로 검색 → place_id, 이름, 주소, 좌표 자동 채움.
  - 지도 표시 (phase 2).
- `bb-admin/src/app/api/places/route.ts`:
  - `GET` — 전체 리스트.
  - `POST` — 신규 등록.
  - `PATCH /[id]` — 수정.
  - `DELETE /[id]` — 삭제.
- bb-app은 "현재 좌표 빠른 등록" 버튼만, 본 관리는 bb-admin.

### 5. 하트비트 엔진

- 서버 측 워커가 location_stack을 주기적으로 (1~5분) 검사.
- 트리거 조건 매칭 시 봇 발화 큐로 푸시 (메시지를 Discord/Ably로 전송).
- 발화 빈도 조절: 같은 종류 트리거는 최소 N분 간격 (스팸 방지).

### 6. 어시스턴트에 전달

- 보이스 요청 시: 봇 컨텍스트에 `현재 위치: {장소이름}` 한 줄 prepend.
- 하트비트 발화 시: 현재 위치 + 직전 위치 + 머문 시간을 컨텍스트로 같이 넘김.
- 좌표 raw 값은 모델에 직접 안 던짐 (디버그 모드만 예외).

## 단계별 작업

1. bb-admin에 places 스토어 + GET/POST API (Google Places autocomplete 포함).
2. bb-admin places 페이지 UI.
3. iOS `LocationService` 추가, Info.plist 키 등록 (When In Use + Always).
4. `VoiceView` 권한 흐름 + 좌표 캡처 (채널 A).
5. `publishRequest` 페이로드 확장.
6. `voice-bridge` places 매칭 로직 + bb-admin API 캐싱.
7. Google Places nearby fallback.
8. iOS significant location change → Ably location 채널 publish (채널 B).
9. 서버 location_stack SQLite 테이블 + Ably 수신 핸들러.
10. 하트비트 엔진: 트리거 조건 + 발화 큐.
11. 어시스턴트 system message에 위치 라인 끼우기.
12. bb-app에 "지금 여기 등록" 단축 버튼 (선택).

## 열린 질문

- 정확도: GPS vs Wi-Fi. 등록 장소 반경 100m면 Wi-Fi로 충분할 가능성.
- 백그라운드 위치 권한을 사용자가 거부할 가능성 — 그 경우 하트비트는 위치 무시하고 시간/캘린더 기반으로만.
- 스택 보관 기간 — 24h가 적당한지, 더 짧게 갈지.
- 하트비트 발화 빈도 디폴트 — 처음엔 보수적으로(하루 2~3회 상한) 시작.
- 좌표 로그를 영구 보관할지 — 분석용으로 일부 남기되 raw가 아닌 place 이름 단위로만.
